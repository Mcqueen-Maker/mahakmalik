Learning Kubernetes the Practical Way: Concepts First, Then EKS

How I overcame Kubernetes intimidation by starting with the official documentation, understanding core concepts first, then diving into hands-on EKS deployment and failure testing.

I'll be honest: Kubernetes had been intimidating me for a while.

Not because I didn't want to learn it ‚Äî but because every Kubernetes tutorial on the internet somehow starts with,

"Assuming you already know Kubernetes‚Ä¶"

Cool. Love that for me.

So this time, I decided to do something boring but effective: I opened the official documentation and started from the very beginning. No shortcuts. No "just deploy this YAML and trust me." Just understanding what the system is actually made of.

And weirdly enough ‚Äî that's when Kubernetes stopped feeling scary.

"Okay, but what is Kubernetes really?"

When I first read the Kubernetes docs, I realized something important: Kubernetes is not a tool you run commands on. It's a system that constantly asks one question:

"Is the cluster in the state it's supposed to be?"

That's it. That's the whole philosophy.

At a high level (straight from the docs), Kubernetes is made up of two big parts:

  • a control plane (the brain)
  • worker nodes (the muscle)

Once that mental model landed, everything else started stacking nicely.

The Control Plane: the overthinker (in a good way)

The control plane doesn't run your app. It just worries about it. Constantly.

It includes:

  • an API server (everything goes through this)
  • a datastore that remembers the desired state
  • schedulers and controllers that keep checking: "Are we good? No? Fixing it."

As a DevOps engineer, this felt familiar. This is automation with anxiety ‚Äî and I mean that as a compliment.

Worker Nodes: where things actually happen

Worker nodes are simpler. They:

  • run containers
  • follow instructions
  • report back

If a node disappears? Kubernetes shrugs and moves on. If a pod dies? Kubernetes replaces it like nothing happened.

This is where the "self-healing" reputation actually makes sense ‚Äî not as magic, but as design.

Pods, Deployments, Services (aka: why YAML exists)

One of the biggest mindset shifts for me was this:

You don't deploy containers. You deploy pods ‚Äî and you usually don't even manage those directly.

Instead:

  • Deployments say how many pods you want
  • Services say how traffic reaches them

You describe what you want. Kubernetes figures out how to keep it true. This is peak DevOps energy.

Why Kubernetes matters for DevOps (the real reason)

Kubernetes isn't popular because it's trendy.

It's popular because:

  • servers fail
  • pods crash
  • traffic spikes
  • humans forget things

Kubernetes assumes failure will happen and builds recovery into the system. For DevOps, that's huge.

It moves you from:

  • "Let me fix this at 3 AM"

to:

  • "The system already fixed it ‚Äî here's the log."

Why I learned Kubernetes through EKS

After the concepts clicked, I went to Amazon Elastic Kubernetes Service ‚Äî because that's what I actually use at work.

EKS handles:

  • control plane availability
  • upgrades
  • scaling the boring (and risky) parts

Which meant I could focus on:

  • clusters
  • nodes
  • deployments
  • behavior under failure

Very DevOps-aligned.

The Lab: "Okay, let's actually do this"

I followed a hands-on lab where the goal was simple: Deploy a web app on EKS and break it on purpose. Honestly? Perfect learning setup.

Step 1: IAM first (because AWS)

Before touching Kubernetes, you create an admin IAM user. This immediately reminded me: Kubernetes doesn't replace cloud security ‚Äî it builds on it.

Step 2: An EC2 "admin" machine

You launch an EC2 instance that becomes your control point. This is where kubectl, eksctl, and AWS CLI all live together and start making sense.

Step 3: Tooling setup (the ritual)

Installing kubectl and eksctl felt like a rite of passage. Once the tools are installed, you verify them:

Verify kubectl installation:

$ kubectl version --client
$ kubectl version --short

Verify eksctl installation:

$ eksctl version

Verify AWS CLI:

$ aws --version

Step 4: Creating the cluster (the fun command)

This single command created a Kubernetes cluster with everything configured:

Create EKS cluster:

$ eksctl create cluster \
  --name my-k8s-cluster \
  --version 1.27 \
  --region us-east-1 \
  --nodegroup-name my-nodes \
  --node-type t3.medium \
  --nodes 2 \
  --nodes-min 1 \
  --nodes-max 3

This single command created:

  • a Kubernetes control plane
  • worker nodes (2 nodes in this case)
  • networking infrastructure
  • autoscaling group (1-3 nodes)

I didn't memorize what it did. I read it. Then watched AWS build it. Much better.

Step 5: Deploying the app

This is where theory met reality. First, verify the cluster was created successfully:

Check cluster status:

$ eksctl get cluster
$ kubectl get nodes
$ kubectl get namespaces

Then I applied a YAML manifest that created a complete deployment:

Sample YAML deployment (app.yaml):

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:latest
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: LoadBalancer
  selector:
    app: nginx
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80

Apply and verify deployment:

$ kubectl apply -f app.yaml
$ kubectl get deployments
$ kubectl get pods
$ kubectl get svc
$ kubectl describe svc nginx-service

You can paste the LoadBalancer external-ip in your browser and it should display the web app. When the load balancer DNS returned the app response, I genuinely smiled.

Step 6: Breaking things on purpose

Then came my favorite part. I terminated both worker nodes manually to test the self-healing capability:

Get node names and terminate them:

$ kubectl get nodes
$ kubectl delete node <node-name-1>
$ kubectl delete node <node-name-2>

Watch Kubernetes automatically replace them:

$ kubectl get nodes -w
$ kubectl get pods
$ kubectl get events --sort-by='.lastTimestamp'

And Kubernetes just‚Ä¶ fixed it.

New nodes came up automatically. Pods were rescheduled to the new nodes. Service kept working without any interruption.

This is when Kubernetes stopped being "concepts" and became trustworthy.

üí° What I actually learned

  • Kubernetes isn't magic ‚Äî it's control loops
  • YAML isn't scary once you understand intent
  • Failure is assumed, not avoided
  • DevOps is about designing for recovery, not perfection

Most importantly:

Kubernetes makes sense when you understand why it behaves the way it does ‚Äî not when you memorize commands.

üîÆ What's next

This is just the beginning. Next up for me:

  • namespaces & RBAC
  • resource limits
  • debugging failed pods
  • deeper EKS networking

But for the first time, Kubernetes doesn't feel like a monster.

It feels like a system I can reason about.

And honestly? That's the best place to start.
